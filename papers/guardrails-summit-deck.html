The Problem

Agents Have Hands

Chatbot

User: "Clean up the project"

Bot: "You can run: rm -rf node_modules"

↑ Suggestion only. User decides.

Agent

User: "Clean up the project"

Agent: [executes: rm -rf /]

↑ Actually does it. No confirmation.

Modern agents can:

  • Read and write files
  • Execute shell commands
  • Call cloud APIs
  • Make network requests
  • Trigger production workflows
Safety is an architectural concern, not a UX feature.
"A chatbot that gives bad advice is annoying. An agent that executes bad advice is dangerous. The moment you give an AI tools — file access, shell, APIs — you need guardrails."

Architecture

Defense in Depth — Three Layers

Layer 3 — Content Safety
Amazon Bedrock Guardrails
Filters harmful content, blocks topics, redacts PII
Layer 2 — Runtime Interception
Hooks (preToolUse / postToolUse)
Inspects and blocks tool calls in real time
Layer 1 — Access Control
Tool Restrictions
Controls which tools exist and what they can access

Layer 1 limits capability

If the tool doesn't exist, the model can't call it.

Layer 2 inspects actions

Every tool call is intercepted before execution.

Layer 3 evaluates content

Input and output checked for harmful or sensitive content.

No single layer is sufficient by itself.
"Think of it like physical security: a locked door (Layer 1), a security camera (Layer 2), and a guard who checks IDs (Layer 3). You need all three."

Layer 1

Tool Restrictions — Least Privilege for Agents

  • Remove dangerous tools by default
  • Start read-only, expand as needed
  • Restrict write access to safe paths
  • Whitelist known-safe shell commands
  • Require human approval for risky tools

Key Principle

If the tool doesn't exist for the agent, the model cannot call it.

// Agent configuration
{
  "tools": ["read", "write", "shell"],

  // Auto-approve only safe tools
  "allowedTools": ["read", "grep"],

  "toolsSettings": {
    "write": {
      "allowedPaths": ["src/**"],
      "deniedPaths":  [".env", "*.key"]
    },
    "shell": {
      "allowedCommands": [
        "npm test",
        "npm run build"
      ]
    }
  }
}
"This is the coarsest but most reliable layer. It's static — set once in config. The agent can only write to src/, can't touch .env or key files, and can only run npm test or npm run build. Everything else is blocked or requires approval."

Layer 2

Runtime Hooks — Real-Time Interception

Agent calls shell("rm -rf /tmp") │ ▼ ┌──────────────────────────┐ │ preToolUse hook runs │ │ │ │ Receives JSON: │ │ • tool_name: "shell" │ │ • command: "rm -rf..." │ │ • session_id, cwd │ │ │ │ exit 0 → ALLOW │ │ exit 2 → BLOCK │ └──────────────────────────┘

Hook use cases:

🛑 Block destructive commands

Reject rm, drop, delete, truncate

🔍 Secret detector

Scan file writes for API keys, tokens

📋 Audit trail

Log every tool call with timestamp

⏱️ Rate limiter

Throttle calls, prevent cost spikes

"Hooks are your real-time interception point. They're just shell scripts that receive JSON on stdin and exit 0 to allow or exit 2 to block. Simple, composable, and powerful. You can stack multiple hooks — audit log + secret scanner + rate limiter."

Layer 3

Amazon Bedrock Guardrails

User prompt │ ▼ ┌─────────────────────────┐ │ Bedrock Guardrails │ │ INPUT check │ └───────────┬─────────────┘ ▼ Foundation Model │ ▼ ┌─────────────────────────┐ │ Bedrock Guardrails │ │ OUTPUT check │ └───────────┬─────────────┘ ▼ Safe response

Six managed safeguards:

  • Content filters — hate, violence, misconduct
  • Denied topics — block specific subjects
  • Word filters — block patterns/terms
  • PII detection — redact SSN, cards, emails
  • Contextual grounding — detect hallucinations
  • Prompt attack filter — block injections
Works on both input and output. Can be applied to any model provider via the ApplyGuardrail API.
"This layer protects at the content level — it understands meaning, not just patterns. It evaluates both what goes into the model and what comes out. And you can use the ApplyGuardrail API independently — even with non-Bedrock models."

Demo 1

Hook Blocks a Destructive Command

1

User asks agent to "clean up temp data"

2

Agent decides to run rm -rf /tmp/data

3

preToolUse hook inspects the command

4

Hook detects rm → exit 2 → BLOCKED

5

Agent adapts: reports the block, suggests alternatives

Safe systems fail visibly and recover gracefully. 
#!/bin/bash
# block-rm.sh — preToolUse hook

INPUT=$(cat)

COMMAND=$(echo "$INPUT" | python3 -c "
import sys, json
d = json.load(sys.stdin)
print(d.get('tool_input', {})
       .get('command', ''))
")

if echo "$COMMAND" | grep -q "rm "; then
    echo "BLOCKED: refusing rm" >&2
    exit 2  # ← BLOCK
fi

exit 0  # ← ALLOW

Even when the model produces unsafe text, the platform catches it before exposure. 
# Create guardrail with PII protection
client.create_guardrail(
  name="agent-safety",

  # Redact sensitive information
  sensitiveInformationPolicyConfig={
    "piiEntitiesConfig": [
      {"type": "EMAIL",
       "action": "ANONYMIZE"},
      {"type": "US_SOCIAL_SECURITY_NUMBER",
       "action": "BLOCK"},
      {"type": "CREDIT_DEBIT_CARD_NUMBER",
       "action": "BLOCK"},
    ]
  },

  # Block harmful content
  contentPolicyConfig={
    "filtersConfig": [
      {"type": "PROMPT_ATTACK",
       "inputStrength": "HIGH"},
    ]
  },
)
